# 设计与调试方法

这套工具不是把现成项目复刻成提示词，而是从三条证据链共同生长出来：已经跑通的流程、后来散落在 AI 对话里的修复、以及没有做成或做得不够好的尝试。

## 1. 从成功经验抽象什么

- 冻结原始输入，再标准化、去重、匹配、评分或生成。
- 批量前先抽样，阶段完成时记录行数、缺失、异常、来源和产物指纹。
- 研究结论拆成事实、推断、建议；事实登记来源、日期、范围和状态。
- 执行者与审查者分开，外部写入前 dry-run，写后 read-back。
- 保存 AI 草稿与人工定稿差异，把真实编辑选择转成规则候选。

这些经验被写进工作流契约、确定性脚本和人工关口，而不是只写在文档里。

## 2. 从散落优化补什么

项目在第一次搭出“工作流头部”后，很多实际修复只发生在聊天窗口。常见变化包括：解析器不兼容时切换路径、写入后再读取核对、匹配时保留原值和依据、渲染后目视检查、输入变化后只重跑受影响分支。

因此运行层采用 append-only 事件日志和输入指纹。中断后从事件恢复；输入改变会使下游失效；产物指纹变化会被 `doctor` 报告。

## 3. 从失败经验增加什么

### 重复重试

失败后再调用同一工具很常见，但不等于恢复。本工具每阶段最多尝试三次，下一次必须声明改变的策略；预算耗尽进入异常队列。只有同一阶段通过预先定义的完成谓词，才叫恢复成功。

### 协议与能力不匹配

模型接口、文件格式、浏览器、连接器、codec、字体和渲染能力不一致，会让“逻辑正确”仍不能完成。因此媒体等 Loop 先建立能力矩阵，并预先写明降级交付。

### 假完成

“命令退出 0”“文件存在”“工具显示成功”都可能是假完成。工作流区分：

- deterministic：schema、指纹、守恒、边界和结构检查；
- assistive：研究、解释、创意和低置信升级；
- human：目视、版权、合规、权重和业务决定；
- mutation：外部副作用。公开首版禁用真实 mutation。

AI 不能给自己的 assistive 产物签发人工批准。

成品审查后又加了一条硬规则：只有 check 已绑定 runner verifier，阶段才允许标为 `deterministic`。目前自动 verifier 只覆盖发布敏感扫描与 plan-only action manifest；预算、权利、事实、媒体质量等尚未实现可靠算法的检查，明确标为 assistive/human，并要求证据文件与职责分离，避免把“清单写得很完整”冒充“软件已经自动验证”。

### 数据假匹配

空字符串、短子串、日期边界和编号格式会造成高置信假匹配。公开方法要求原值/规范值双轨、稳定键、匹配依据、置信度、异常队列、边界样本和前后守恒。

### 内容过度断言

未经核实的数字、功能、因果和单一信源会让内容看似完整但不能发布。`research-to-brief` 和 `content-repurpose` 强制声明台账、冲突保留和编辑独立复核。

### 过度自动化

自动剪辑可能损害语境，平台批量操作可能违规或触发风控。`media-production` 默认先给编号时间线，由人选 KEEP/CUT；所有平台动作只到 plan-only 清单。

## 4. 为什么使用 typed DAG

真实项目不是一条直线：研究和数据可以并行，输入变化只影响部分产物，人工可能拒绝，渲染可能降级，外部动作可能部分成功。工作流用有类型 DAG 描述依赖，用事件重放得到状态，用内容哈希关联产物。

当前公开运行器不执行真实连接器，避免在幂等、租户、审批、版本再校验和补偿尚未逐连接器验证前制造副作用。

## 5. 新失败如何沉淀

每次人工修复后，不直接改“总提示词”。先生成一个最小匿名失败样例，写清：

1. 触发输入；
2. 预期不变量；
3. 实际失败；
4. 改变的策略；
5. 通过的确定性或人工证据；
6. 是否扩大权限或产生副作用。

然后把它加入回归测试或规则提案。规则提案只有人工审批、schema 检查、代码审查和回归测试通过后才能成为正式规则。
